# Техническое задание (финальная консолидированная версия, актуализированная)

## Проект: специализированная CRM/PWA для Katet.tech

## Формат: MVP для реализации в Codex

## Статус документа: финальная рабочая версия (актуализировано под текущий план)

---

## 1. Цель проекта

Разработать self-hosted web-only CRM/PWA для Katet.tech, которая заменяет Bitrix24 в ключевом процессе аренды спецтехники:

**Лид → Заявка → Бронь → Выезд → Завершение**

Система не является универсальной CRM и проектируется под операционный процесс аренды спецтехники с обязательным управлением лидами, заявками, бронями, подрядчиками и завершением заказа.

---

## 2. Утверждённая продуктовая логика MVP

### 2.1 Базовая модель

- UX-модель: **lead-centric**.
- При этом в домене обязательно разделяются сущности:
  - `Lead`
  - `Application`
  - `Reservation`
  - `Departure`
  - `Completion`

### 2.2 Основная воронка (kanban)

Обязательные стадии:

1. Лид
2. Заявка
3. Бронь
4. Выезд
5. Завершено
6. Некачественный

### 2.3 Ключевые правила процесса

- Один `Lead` может иметь только **одну активную** `Application`.
- `Application` поддерживает несколько позиций (`ApplicationItem`).
- Перевод лида в заявку выполняется после связи менеджера с клиентом.
- Внутри стадии «Бронь» обязателен подпроцесс выбора источника исполнения:
  - **своя техника**;
  - **подрядчик**.
- Для **своей техники** рабочий сценарий MVP:
  - выбор типа техники;
  - при возможности уточнение до конкретной единицы;
  - проверка конфликта;
  - подтверждение брони;
  - перевод в готовность к выезду.
- Для **подрядчика** рабочий сценарий MVP:
  - выбор подрядчика;
  - запрос/получение подтверждения;
  - фиксация условий по конкретной броне;
  - подтверждение брони;
  - перевод в готовность к выезду.
- Для подрядной техники **отсутствие конкретной единицы техники у подрядчика не должно блокировать бронь**.
- Уточнение до модели/единицы подрядчика допускается позже и является **опциональным**.

---

## 3. Утверждённые требования MVP (обязательно в первой версии)

### 3.1 Лиды и входящие

Система должна принимать лиды из:

- сайта;
- Mango;
- Telegram;
- MAX;
- ручного ввода менеджером.

Также обязательно:

- kanban-представление;
- список лидов;
- карточка лида;
- поиск дублей по телефону и названию компании.

### 3.2 Заявки

- Перевод `Lead → Application` после контакта менеджера.
- Конструктор заявки с несколькими позициями.
- Редактирование позиций заявки в рамках одной карточки.

### 3.3 Бронь

- Бронь ставится на **тип техники**.
- Позже бронь может быть уточнена до **конкретной единицы** — прежде всего для **своей техники**.
- Конфликт брони показывается как **warning**, без hard-block.
- Бронь снимается:
  - вручную менеджером;
  - автоматически при переводе заявки в `completed` или `unqualified`.

#### Дополнительные правила MVP для источников исполнения

- Если источник = **своя техника**, система должна поддерживать:
  - выбор типа техники;
  - выбор конкретной единицы техники;
  - soft conflict warning по пересечению интервалов.

- Если источник = **подрядчик**, система должна поддерживать:
  - выбор подрядчика;
  - статус подтверждения подрядчиком;
  - комментарий/заметку по условиям;
  - возможность перевести бронь дальше **без заполненного unit подрядчика**.

Текущий backend/QA срез:

- Внутренняя готовность брони может учитывать подрядчика без собственной единицы техники.
- Создание `Departure` через `POST /departures` сейчас требует выбранный `equipmentUnitId` для брони (`QA-REQ-014`).
- Не менять это поведение без отдельного продуктового решения, потому что оно влияет на тестовый контракт.

#### Минимальные статусы подтверждения подрядчика

- `not_requested`
- `requested`
- `confirmed`
- `declined`
- `no_response`

### 3.4 Подрядчики, техника и справочники

- Подрядчики — постоянная часть процесса MVP.
- Нужен справочник **типов техники**.
- Нужен справочник **единиц своей техники**.
- Нужен справочник **подрядчиков**.
- В карточке подрядчика в MVP должны храниться:
  - название;
  - контакты;
  - регионы/зона работы;
  - категории/типы техники;
  - заметки по условиям;
  - история использования в бронях.
- Нужен быстрый выбор подрядчика в позиции/броне.
- **Детальный учёт парка подрядчика не является обязательным требованием MVP.**
- Карточки конкретных единиц техники подрядчика, их доступность и conflict-control по unit подрядчиков допускаются как **опциональное расширение post-MVP** или для ограниченного числа стратегических подрядчиков.

### 3.5 Выезд и завершение

- Есть стадия «Выезд».
- Завершение заказа в MVP — **ручное действие менеджера**.

### 3.6 Клиенты

Карточка клиента должна включать:

- реквизиты;
- контакты;
- историю лидов;
- историю заявок;
- любимые категории техники;
- комментарии.

Повторный заказ обязателен:

- из карточки клиента;
- из прошлого заказа.

### 3.7 Журнал действий

Обязателен системный журнал действий по ключевым операциям:

- создание;
- редактирование;
- смена стадии;
- установка/снятие брони;
- завершение заявки.

### 3.8 Базовая аналитика MVP

Обязательные отчёты:

- скорость обработки;
- конверсия в заявку;
- потерянные лиды;
- зависшие лиды.

## 4. Роли и доступ

### 4.1 Роли MVP

- `Admin`
- `Manager`

### 4.2 Принципы доступа

- Отдельной роли диспетчера в MVP нет.
- Все основные действия выполняет менеджер.
- Ограничения доступа реализуются на backend-уровне (server-side authorization).
- Режимы доступа:
  - менеджер видит только свои записи;
  - админ видит всё.

## 5. Доменная модель MVP (консолидированно)

### 5.1 Основные сущности

- `Lead`
- `Application`
- `ApplicationItem`
- `Reservation`
- `Client`
- `EquipmentType`
- `EquipmentUnit`
- `Subcontractor`
- `Departure`
- `ActivityLog`
- `IntegrationEvent`

### 5.2 Ключевые связи

- один `Lead` → одна активная `Application`;
- одна `Application` → много `ApplicationItem`;
- одна `ApplicationItem` → одна активная `Reservation`;
- `Reservation` может ссылаться:
  - на тип техники;
  - на конкретную единицу **своей техники**;
  - на выбранного `Subcontractor`;
- один `Client` → много лидов/заявок;
- один `Subcontractor` → много броней, категорий техники и записей истории использования;
- связь `Subcontractor → EquipmentUnit` в MVP **не является обязательной**;
- любая ключевая сущность → записи в `ActivityLog`.

### 5.3 Статусы

#### Kanban

- `lead`
- `application`
- `reservation`
- `departure`
- `completed`
- `unqualified`

#### Внутренние стадии брони

- `needs_source_selection`
- `searching_own_equipment`
- `searching_subcontractor`
- `subcontractor_selected`
- `type_reserved`
- `unit_defined`
- `ready_for_departure`
- `released`

#### Уточнение по использованию внутренних стадий

- Для сценария **своя техника** ключевыми являются:
  - `searching_own_equipment`
  - `type_reserved`
  - `unit_defined`
  - `ready_for_departure`

- Для сценария **подрядчик** ключевыми являются:
  - `searching_subcontractor`
  - `subcontractor_selected`
  - `type_reserved`
  - `ready_for_departure`

- Стадия `unit_defined` для подрядчика **не обязательна**.

### 5.4 Обязательные поля брони по сценарию исполнения

#### Для своей техники

- `equipmentTypeId`
- `equipmentUnitId` — если известен
- `plannedStart`
- `plannedEnd`

#### Для подрядчика

- `equipmentTypeId`
- `subcontractorId`
- `confirmationStatus`
- `plannedStart`
- `plannedEnd`
- `comment` — опционально
- `promisedModelOrUnit` — опционально

### 6.1 Обязательные интеграции

- Сайт
- Mango

- webhook endpoints и integration log.

## 7. Миграция данных (финально зафиксировано)

Для устранения противоречий принимается единый вариант:

- переносим **всех клиентов**;
- переносим **все контакты**;
- переносим **активные и релевантные лиды**;
- старые закрытые лиды остаются в старой системе как архив.

Импорт в MVP:

- CSV;
- сопоставление колонок;
- предпросмотр перед импортом;
- базовая проверка дублей;
- лог импорта.

## 8. Технический подход и архитектурные рекомендации (не расширяют бизнес-scope MVP)

### 8.1 Архитектурный стиль (рекомендация)

- Модульный монолит.

### 8.2 Стек

#### 8.2.1 Фактически принятый стек frontend (MVP-прототип)

По факту разработки MVP-прототипа UX был принят стек:

- `Vite + React 18 + TypeScript`
- UI: **`shadcn/ui + Radix + Tailwind CSS 4`** (основной слой компонентов)
- State shell: `React Context` (`layoutStore`) + частичный URL sync (`routeSync`)
- DnD: `react-dnd`
- Данные: моки (пока нет backend)

Причина: MVP UX собирается из Figma Make прототипа, где целевой стек — Vite + shadcn, что ускоряет итерации по UX.
`MUI` установлен, но используется минимально. Миграция на `Next.js + MUI` допустима post-MVP при появлении требований по SSR/SEO/серверному middleware.

#### 8.2.2 Рекомендуемый стек backend (ещё не реализован)

- Backend: `NestJS` (допустим эквивалентный server-layer подход, например Fastify + tsyringe)
- DB: `PostgreSQL`
- ORM: `Prisma`
- Auth: `JWT + refresh token`
- Server state на клиенте: `@tanstack/react-query`
- UI state на клиенте (при расширении): `Zustand` (опционально, текущий `layoutStore` покрывает shell)
- Deploy: `Docker Compose + Nginx`
- PWA: service worker + installable manifest (post-MVP, если потребуется installable)

Допустимая альтернатива UI-библиотеки для backoffice-плотных таблиц в будущем: `MUI DataGrid` — не основной вариант первой версии.

### 8.3 Рекомендации по интеграционной надёжности (NFR)

`Retry / replay / расширенное логирование integration events` рекомендуется как технический контур надёжности, но не расширяет обязательный бизнес-scope первой версии, если это критически усложняет MVP.

## 9. Out of scope для MVP

В первой версии **не реализуется**:

- обязательный контур plan/fact работ;
- ручной учёт факта смен, итоговых сумм, итогового количества смен;
- частичные оплаты;
- отдельный финансовый модуль;
- бухгалтерия / 1С;
- ЭДО внутри системы;
- мобильное приложение водителя;
- телематика;
- рейтинг подрядчиков;
- чёрный список подрядчиков;
- отдельная роль диспетчера;
- отдельные сущности `Object / Site / ContactPerson`;
- кабинет клиента;
- кабинет подрядчика;
- **обязательный детальный учёт парка подрядчиков**;
- **обязательные карточки unit подрядчиков**;
- **расписание и conflict-control по unit подрядчиков**;
- **углублённая аналитика подрядного парка**.

## 11. Критерии приёмки MVP (консолидированно)

MVP считается принятым, если:

1. менеджер может создать лид вручную;
2. лиды поступают из сайта, Mango, Telegram, MAX;
3. kanban содержит 6 утверждённых стадий;
4. лид переводится в заявку после контакта менеджера;
5. из одного лида нельзя создать две активные заявки;
6. заявка поддерживает несколько позиций;
7. в позиции можно выбрать сценарий исполнения:
   - своя техника;
   - подрядчик;
8. бронь ставится на тип техники;
9. для **своей техники** бронь может уточняться до единицы;
10. для **подрядчика** бронь может двигаться дальше **без unit**, если выбран подрядчик и зафиксирован статус подтверждения;
11. конфликт брони отображается warning;
12. бронь снимается вручную и при завершении/некачественном исходе;
13. есть стадия «Выезд» и ручное завершение;
14. есть карточка клиента и повторный заказ;
15. работает поиск дублей по телефону/компании;
16. работает журнал действий;
17. доступны базовые отчёты: скорость обработки, конверсия в заявку, потерянные и зависшие лиды.

## 12. Состояние реализации (актуализация 2026-05-06)

### 12.1 Сделано

- Backend реализован в `app/backend` (NestJS + Prisma + PostgreSQL + миграции + seed).
- Работает auth/RBAC контур: `POST /auth/login`, `POST /auth/refresh`, `GET /auth/me`, guards и роли `admin/manager`.
- Реализованы API-модули MVP: `leads`, `clients`, `applications`, `application-items`, `reservations`, `departures`, `completions`, `tasks`, `directories`, `integrations`, `imports`, `activity`, `stats`, `users`, `settings`, `navigation`, `health`.
- Ключевые бизнес-инварианты enforced на уровне БД/сервисов:
  - одна активная заявка на лид;
  - одна активная бронь на позицию;
  - conflict-warning по бронированиям (soft warning);
  - дедуп (телефон/компания) в create/import сценариях.
- Frontend API wiring покрывает основные домены: Leads, Applications, Reservations, Clients, Departures, Completion.
- Home/Control подключены к `/stats` и `/activity/search`; My Tasks работает API-first и поддерживает write-сценарии (`create/update/status/duplicate/archive/subtasks`).
- Для Home/My Tasks поддержана очистка дедлайна end-to-end (`dueDate: null`, включая совместимость с пустой строкой на входе).
- Admin: Imports (`preview/run/report`) и Integrations (`list/detail/retry/replay`) подключены в API-режиме.
- Control analytics переведён на server-side контракт `GET /stats/analytics?viewId=...&sampleTake=...`.
- Закрыт cross-module dead-control аудит для клиентских CTA/link точек в Reservation/Departure/Completion (включая API workspace-варианты) и shared affordance (`DenseDataTable`, `KpiRow`).
- Testing reset 05.05.2026: прежние smoke/e2e команды и результаты удалены.
- Текущая rebuilt validation модель строится от `QA_REQUIREMENTS.md`: backend API contract/integration/coverage, frontend adapter coverage, browser e2e smoke/full gate.

### 12.2 Не сделано / оставшиеся разрывы до полного MVP

- Расширение форматов импорта не в приоритете (фокус на hardening текущего CSV-flow).
- Import/integration negative scenarios требуют дополнительного contract/browser покрытия после уточнения требований.
- Production-like signed webhook fixtures и CSV size/error taxonomy требуют отдельного hardening.
- `leads.service.ts` сохраняет legacy/compatibility auto-create Departure при переходе Lead `reservation -> departure`; текущий QA/API путь считает перевод в выезд ручным действием через `POST /departures`.
- PWA (service worker/manifest) не является реализованной целью текущего среза.

### 12.3 Краткий план доведения до MVP (по шагам)

1. **Усилить import/integration hardening.** Добить backend CSV safety limits/error taxonomy и CI-проверку signed webhook fixtures с production-like secret profiles.
2. **Расширять coverage только от подтверждённых требований.** Новые browser/API checks добавлять после фиксации ожиданий в `QA_REQUIREMENTS.md`.
3. **Поддерживать rebuilt quality gates.** При изменениях запускать релевантные команды из `TESTING_STRATEGY.md` и обновлять `docs/TEST_EXECUTION_REPORT.md`.
